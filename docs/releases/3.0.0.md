# ADAAD v3.0.0 — Phase 5 GA (Multi-Repo Federation)

**Released:** 2026-03-06  
**Tag:** `v3.0.0`  
**Version file:** `VERSION` (`3.0.0`)

## Scope summary

v3.0.0 promotes Phase 5 capabilities to GA — extending ADAAD from single-repo mutation governance to governed cross-repo evolution with deterministic federation replay, HMAC-hardened key validation, and a federated evidence matrix that gates every cross-repo acceptance decision.

This is a major version release. The governance constitutional model is unchanged; all Phase 5 additions operate within and extend the existing constitutional envelope.

## Phase 5 deliverables

### PR-PHASE5-01 — HMAC Key Validation (M-05)
- `runtime/governance/federation/key_registry.py` enforces minimum-length HMAC key at boot (fail-closed)
- Boot halts with `FederationKeyError` if key material is absent or below threshold
- `tests/governance/federation/test_federation_hmac_key_validation.py` — 100% branch coverage

### PR-PHASE5-02 — Cross-Repo Lineage (`federation_origin`)
- `LineageLedgerV2` extended with `federation_origin: FederationOrigin | None` field
- Mutations carrying federated origin are traceable to source-repo epoch chain
- `tests/test_lineage_federation_origin.py` — replay-stable; origin preserved across serialization round-trips

### PR-PHASE5-03 — FederationMutationBroker
- `runtime/governance/federation/mutation_broker.py` — governed cross-repo mutation propagation
- `GovernanceGate.approve_mutation()` required in BOTH source and destination repos before any federated mutation is accepted
- Fail-closed: any gate failure in either repo rejects the federated proposal unconditionally
- `tests/test_federation_mutation_broker.py`

### PR-PHASE5-04 — FederatedEvidenceMatrix
- `runtime/governance/federation/federated_evidence_matrix.py` — cross-repo determinism verification gate
- Release gate output now includes `federated_evidence` section; a non-zero `divergence_count` blocks promotion
- `tests/test_federated_evidence_matrix.py`

### PR-PHASE5-05 — EvolutionFederationBridge + ProposalTransportAdapter
- `runtime/governance/federation/evolution_federation_bridge.py` — lifecycle wiring: broker + evidence matrix initialised and torn down with `EvolutionRuntime`
- `runtime/governance/federation/proposal_transport_adapter.py` — flush/receive proposals via `FederationTransport`
- `tests/test_evolution_federation_bridge.py`, `tests/test_proposal_transport_adapter.py`

### PR-PHASE5-06 — Federated Evidence Bundle + Release Gate
- `runtime/evolution/evidence_bundle.py` extended: `federated_evidence` section emitted in release gate output
- Release gate fails if `federated_evidence.divergence_count > 0`
- `tests/test_evidence_bundle_federated.py`

### PR-PHASE5-07 — Federation Determinism CI + HMAC Key Rotation Runbook
- `.github/workflows/federation_determinism.yml` — Phase 5 CI gate: verifies cross-repo federated mutation determinism on every PR touching federation runtime
- Constitutional requirement enforced: 0 divergences per federated epoch
- `docs/runbooks/hmac_key_rotation.md` — operational runbook for HMAC key rotation in production federation deployments

## Breaking changes

- `LineageLedgerV2` serialization schema version incremented; ledgers written by v2.x are forward-compatible (null `federation_origin`) but v3.0 ledgers are not backward-compatible with v2.x readers that do not handle the new field.
- `evidence_bundle` release gate output schema includes new `federated_evidence` section; consumers parsing release gate JSON must handle the new key.

## Migration notes

1. Ensure `ADAAD_FEDERATION_HMAC_KEY` environment variable is set before deploying any federation-enabled node; boot will fail-closed if absent.
2. Update any release gate JSON consumers to handle the `federated_evidence` section.
3. Existing single-repo deployments require no changes; all federation paths are opt-in via transport configuration.
